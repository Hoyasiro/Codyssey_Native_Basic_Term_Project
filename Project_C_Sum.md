# [Project C] AI 기반 소셜 미디어 콘텐츠 자동 생성 — 결과물 해설서

> **한 줄 요약**
> "주제 하나를 넣으면 → 인스타·블로그·X용 글이 각각의 톤으로 알아서 써지고 → 대표 이미지까지 붙어 → 노션에 정리되는 시스템"

---

## 1. 이 미션의 본질부터 짚고 갑니다

미션 소개(§1)의 핵심 문장입니다.
> "하나의 주제를 입력하면 여러 소셜 미디어 플랫폼에 맞는 **텍스트와 이미지를 자동으로 생성**하는 워크플로우를 팀 단위로 구축합니다."

여기서 승부처는 **"플랫폼별 특성 반영"** 입니다.
같은 글을 세 군데에 복붙하면 이 과제는 존재 이유가 없어집니다.
**인스타는 짧게 + 해시태그, 블로그는 길게 + 소제목, X는 글자 수 제한.**
이 차이를 **프롬프트로 통제했다는 증거**를 남기는 것이 채점의 중심입니다.

구조는 Project B와 유사한 '자동화'지만, **콘텐츠 기획 감각이 함께 요구되는 하이브리드 과제**입니다.

---

## 2. 제출해야 하는 결과물 3종 (근거: §2 최종 결과물)

### 산출물 ①: 자동화 워크플로우

**요구 조건**

- 주제 입력 시 플랫폼별 콘텐츠가 **자동 생성**되는 워크플로우
- 워크플로우 **구조 스크린샷**
- **각 단계별 역할과 연결 구조**를 설명하는 문서

**예시 — 미션 §7이 제시한 구조**

```
[1] 주제 입력  ── 예: "2026년 AI 트렌드"
        ↓
[2] 플랫폼별 텍스트 생성
      ├─ 인스타그램용 캡션
      ├─ 블로그용 본문
      └─ X용 요약
        ↓
[3] 이미지 생성  ── 주제에 맞는 대표 이미지
        ↓
[4] 노션 저장   ── 플랫폼별 결과물 정리
```

**단계별 설명 문서 작성 예시**

| 노드 | 도구 | 역할 |
|---|---|---|
| Form Trigger | n8n Form | 주제 키워드 1개 입력받음 |
| LLM ×3 | OpenAI API | 플랫폼별 프롬프트로 각각 호출 |
| Image Gen | DALL·E 3 | 주제 기반 대표 이미지 1장 |
| Notion | Notion API | 플랫폼별로 구분해 저장 |

---

### 산출물 ②: 플랫폼별 콘텐츠 결과물

**요구 조건 (§2, §4)**

- **최소 3개 플랫폼**에 맞는 텍스트 (예: 인스타그램 / 블로그 / 트위터)
- 콘텐츠에 활용할 **대표 이미지 1장 이상**
- 결과물이 저장된 **노션 등 협업 도구 페이지**
- 그 페이지 안에 **프로젝트 개요 + 팀 구성원 역할 + 개인별 작업 요약** 포함

**플랫폼별 정량 기준 (§4 기능 요구 사항에 명시된 예시)**

| 플랫폼 | 형식 요구 | 톤앤매너 예시 |
|---|---|---|
| 인스타그램 | 짧은 캡션 + **해시태그 10개 내외** | 가볍고 감각적, 이모지 허용 |
| 블로그 | **500자 이상** 본문 + **소제목 구조** | 정보 전달 중심, 설명형 |
| X (구 트위터) | **280자 이내** 요약 | 핵심만, 단정적 문장 |

**결과 예시 — 주제: "2026년 AI 트렌드"**

```
[인스타그램]
캡션: AI가 바꾸는 일상, 올해 가장 주목할 트렌드 5가지를 정리했어요 ✨
해시태그: #AI트렌드 #인공지능 #생성형AI #테크 #자동화 #에이전트 ...(10개 내외)

[블로그]
제목: 2026년 반드시 알아야 할 AI 트렌드 5가지
## 1. 멀티모달 AI의 일상화
   (본문...)
## 2. 에이전트 자동화의 본격화
   (본문...)
→ 총 500자 이상

[X]
2026년 AI 핵심 트렌드: 멀티모달 확산, 에이전트 자동화 본격화,
온디바이스 AI 상용화. 올해 놓치면 안 될 변화들. #AI #트렌드
→ 280자 이내
```

> ⚠️ **"저장이 플랫폼별로 구분되어야 한다"(§4 결과 저장)** 는 조건이 있습니다.
> 노션에 세 글을 한 페이지에 죽 이어붙이지 말고,
> **`플랫폼` 속성(Select: Instagram / Blog / X)** 을 두고 **행을 나눠 저장**하는 편이 요구사항에 정확히 부합합니다.

---

### 산출물 ③: 프롬프트 템플릿 문서

**요구 조건 (§2)**

- **플랫폼별 프롬프트 설계 내용**
- 프롬프트 **수정 과정 기록 (초안 → 수정 → 최종)**

**작성 예시 — 인스타그램용 프롬프트**

| 단계 | 프롬프트 | 변경 이유 | 결과 차이 |
|---|---|---|---|
| 초안 | `{주제}에 대해 인스타그램 글 써줘` | — | 블로그처럼 길게 나옴. 해시태그 없음 |
| 수정 1 | `{주제}에 대한 인스타그램 캡션. 3문장 이내, 친근한 반말체, 해시태그 10개 포함` | 분량·톤·해시태그 수를 강제해야 형식이 안정됨 | 형식은 맞으나 해시태그가 영어/한글 뒤섞임 |
| 최종 | 위 + `해시태그는 모두 한글로. 이모지는 1개만. 출력은 JSON: {"caption": "", "hashtags": []}` | 후속 노드에서 파싱해야 하므로 출력 구조 고정 필요 | 노션 자동 매핑 성공, 결과 일관성 확보 |

> 💡 **JSON 출력 강제**는 자동화 과제에서 사실상 필수 테크닉입니다.
> 자유 문장으로 받으면 노션 속성에 매핑할 때 매번 깨집니다.

## 3. 놓치기 쉬운 세부 요구사항 (근거: §4)

| 항목 | 정확한 요구 |
|---|---|
| 입력/트리거 | 주제·키워드를 **입력받는 방식 구현** (폼, 수동 입력 등) |
| 트리거 동작 | 입력 시 워크플로우가 **자동 트리거** |
| 텍스트 | 최소 **3개 플랫폼**, 각 플랫폼 특성 반영 |
| 톤앤매너 | **플랫폼별로 적절히 조정**되어야 함 |
| 이미지 | **소셜 미디어에 활용 가능한 품질** |
| 저장 | 노션 등에 **자동 저장**, **플랫폼별 구분** |

## 4. 반드시 피해야 할 감점 요소 (근거: §6 제약 사항)

Project C의 제약은 짧지만 무겁습니다.

| 금지 사항 | 설명 |
|---|---|
| 🚫 **저작권 문제가 있는 콘텐츠를 실제 SNS에 무단 게시** | 생성 결과물을 실계정에 올릴 계획이라면 특히 주의 |
| 🚫 **타인의 콘텐츠를 복사하여 제출** | 기존 브랜드 캡션·블로그 글의 복붙 금지 |

> 💡 안전한 운영 방법: **실제 게시는 하지 않고**, 노션에 "발행 예정 콘텐츠" 형태로만 저장하면 리스크가 사라집니다.
> 굳이 게시한다면 **팀 테스트 계정 + 전면 비공개** 설정을 권합니다.

## 5. 보너스 과제 (근거: §5)

- **보너스 1 – A/B 테스트**
  - 같은 주제로 **톤앤매너가 다른 2가지 버전** 생성 (예: 친근한 버전 vs 전문적인 버전)
  - **플랫폼·주제 등 변수별로 어떤 콘텐츠가 효과적인지 결론 정리**

**정리 예시**

| 변수 | 친근한 톤 | 전문적인 톤 | 결론 |
|---|---|---|---|
| 인스타그램 | 이모지·구어체가 자연스러움 | 딱딱하고 이질적 | 인스타는 친근 톤 우세 |
| 블로그 | 정보 신뢰도 낮아 보임 | 소제목 구조와 잘 맞음 | 블로그는 전문 톤 우세 |
| X | 짧아서 톤 차이 미미 | 단정적 어투가 유리 | 주제가 기술이면 전문 톤 |

## 6. 팀 작업 분배 & 40시간 운용 제안

| 역할 | 담당 업무 | 예상 시간 |
|---|---|---|
| 워크플로우 빌더 | 폼 트리거, 노드 연결, 노션 연동 | 12h |
| 프롬프트 설계 ×2 | 플랫폼별 프롬프트 튜닝 + 템플릿 문서 | 14h |
| 이미지 담당 | 대표 이미지 생성·품질 확인 | 6h |
| 문서/노션 | 프로젝트 개요, 팀 역할, 개인별 작업 요약 정리 | 8h |

> 💡 **팀원 3~4명이면 "1인 1플랫폼" 분배가 가장 자연스럽습니다.**
> 각자 자기 플랫폼의 프롬프트를 초안→수정→최종까지 책임지면 §2의
> "개인별 작업 요약" 항목도 자동으로 채워집니다.
