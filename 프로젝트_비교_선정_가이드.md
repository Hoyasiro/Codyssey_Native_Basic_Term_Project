## 3. 난이도 & 리스크 비교

*10점 만점, 숫자가 클수록 부담이 큼*

| 항목 | A | B | C |
|---|:---:|:---:|:---:|
| 기술 진입장벽 (API·노드 연결) | 3 | **8** | 7 |
| 도구 학습량 | 5 | 7 | 7 |
| 디자인/미적 감각 요구 | **9** | 1 | 4 |
| 글쓰기·기획 감각 요구 | 4 | 3 | **8** |
| 요구사항 명확도 (정답 경로가 뚜렷한가) | 6 | **9** | 7 |
| "안 돌아가서 망할" 리스크 | 2 | **8** | 6 |
| 유료 비용 발생 가능성 | 6 (이미지 생성 크레딧) | 5 (API 호출) | **7** (텍스트+이미지 둘 다) |
| 팀 분업 용이성 | 5 | 6 | **8** |

### 해석

- **A**: 기술 리스크는 가장 낮지만, **결과물의 완성도가 곧 눈으로 평가**됩니다. 스타일 일관성(§3 과제 목표)이 안 맞으면 바로 티가 납니다.
- **B**: **가장 어렵지만 가장 명확**합니다. 요구사항이 숫자로 못 박혀 있어(3줄 요약, 재시도 2회, 중복 방지 키 등) 체크리스트만 지키면 감점 여지가 적습니다. 대신 **API 연동이 막히면 팀 전체가 멈춥니다.**
- **C**: B의 자동화 골격에 **콘텐츠 품질 판단**이 얹힙니다. 분업이 가장 쉽고(1인 1플랫폼), 겉보기 산출물도 풍성합니다.

## 5. 팀 유형별 추천 시나리오

### 시나리오 ①: "우리는 비개발자 위주다"
→ **Project A 권장**
API 연동 없이 이미지 생성 툴과 Figma만으로 완결됩니다. 실패 리스크가 가장 낮습니다.
단, §6 제약(Pinterest·Dribbble 캡처 제출 엄격 금지, 출처 필수 기록)을 팀 전체에 반드시 공지하세요.

### 시나리오 ②: "자동화를 제대로 배우고 싶다"
→ **Project B 권장**
스케줄 트리거 / 필터링 / AI 가공 / 저장 / 에러 처리까지 **자동화의 전 요소**가 들어 있습니다(§1, §4).
이후 어떤 업무 자동화로도 확장 가능한 표준 구조를 익히게 됩니다.

### 시나리오 ③: "결과물이 화려했으면 좋겠다 + 분업이 필요하다"
→ **Project C 권장**
텍스트 3종 + 이미지가 한 번에 나오므로 시연이 인상적이고, 플랫폼별로 담당을 쪼개기 좋습니다.
보너스(A/B 테스트, §5)까지 하면 발표 분량이 넉넉해집니다.

### 시나리오 ④: "실력이 섞여 있다 (개발 1~2명 + 비개발 다수)"
→ **Project C 권장**
개발 인원이 워크플로우 골격을 잡고, 나머지가 프롬프트·콘텐츠·문서를 맡는 구조가 자연스럽습니다.

**② 팀 회의 (60분)**
1. 각자 선호 1순위와 **이유 1문장**만 말합니다. (10분)
2. 아래 **리스크 점검**을 함께 확인합니다. (20분)
   - 이미지 생성 툴 유료 크레딧을 낼 사람이 있는가? *(A·C 해당)*
   - OpenAI API 키를 발급·결제할 사람이 있는가? *(B·C 해당)*
   - 노션 워크스페이스를 팀으로 쓸 수 있는가? *(B·C 해당)*
   - Figma 계정과 기본 사용법을 아는 사람이 있는가? *(A 해당)*
3. **결선 투표**: 상위 2개만 남기고 다시 투표합니다. (10분)
4. 동점이면 → **"우리 팀이 40시간 안에 확실히 끝낼 수 있는 쪽"** 을 기준으로 재논의합니다. (20분)

**③ 결정 직후 즉시 할 일**
- 선정 프로젝트의 **체크리스트를 그대로 팀 노션에 복사**
- 역할 배정표 작성 (세 프로젝트 모두 **개인별 작업 요약**을 요구합니다 — A §2 작업 로그, B §2 README, C §2 노션 페이지)
