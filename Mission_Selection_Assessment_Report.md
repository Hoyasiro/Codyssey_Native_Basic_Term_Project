## 📌 결론 먼저

| 순위 | 프로젝트 | 종합 점수 | 판정 |
|:---:|---|:---:|---|
| **1** | **Project A — AI 기반 UI/UX 디자인 시안 제작** | **84 / 100** | ✅ **권고** |
| 2 | Project C — 소셜 미디어 콘텐츠 자동 생성 | 47.5 / 100 | ⚠️ 조건부 |
| 3 | Project B — 뉴스 요약 자동화 워크플로우 | 43.5 / 100 | ❌ 비권장 |

**핵심 근거 세 줄**

1. **선호도에서 A만 유일하게 '반대표(1~2점)가 0명'** 입니다. B와 C는 각각 1점을 준 인원이 1명씩 있습니다.
2. **A는 산출물 3종의 담당자가 이미 전원 자원한 상태**이지만, B·C는 필수 산출물의 담당 공백이 존재합니다.
3. 팀의 개발자 비율이 **25%(1/4)** 이고 자동화 툴 경험자는 **0명**입니다. 가이드 §3의 '기술 진입장벽'(A 3점 / B 8점 / C 7점) 기준에서 A가 압도적으로 유리합니다.

### 1-2. 팀 역량 프로필 (객관식 4문항)

| 문항 | 네 | 아니오 / 기타 | 팀 상태 |
|---|:---:|:---:|---|
| 개발자입니까? | **1명 (25%)** | 3명 (75%) | 비개발자 중심 팀 |

> 📊 **읽는 법**
> 세 역량이 모두 "1 : 3" 구조입니다. 즉 **어떤 프로젝트를 골라도 특정 1명에게 기술이 몰립니다.**
> 그렇다면 판단 기준은 "그 1명이 막혔을 때 나머지 3명이 계속 일할 수 있는가"가 됩니다. (가이드 §3 *"안 돌아가서 망할 리스크"* 항목)

### 1-3. 선호도 집계

| 프로젝트 | 1점 | 2점 | 3점 | 4점 | 5점 | **평균** | 반대표(1~2점) |
|---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **A** | 0 | 0 | 1 | 2 | 1 | **4.00** | **0명** ✅ |
| B | 1 | 1 | 1 | 0 | 1 | 2.75 | 2명 ⚠️ |
| C | 1 | 1 | 1 | 0 | 1 | 2.75 | 2명 ⚠️ |

> 💡 B와 C는 평균이 정확히 동률(2.75)이지만, **분포가 양극단으로 갈라져 있습니다.**
> 5점을 준 사람과 1점을 준 사람이 공존한다는 것은, 선택 시 **일부 팀원이 낮은 동기로 40시간을 보내게 된다**는 뜻입니다.
> 반면 A는 **최저점이 3점**으로, 전원이 최소한 중립 이상입니다. 다수결 관점에서 A가 가장 안전한 합의점입니다.

## 4. 평가 ② — 가이드 §4 판단 질문 대입 (배점 30점)

가이드 §4의 9개 질문에 설문 결과를 그대로 대입했습니다.

| 가이드 §4 질문 | 설문 근거 | 답 | 가이드가 지목한 프로젝트 |
|---|---|:---:|---|
| 팀에 Figma를 만져본 사람이 있다 | "네" 1명 (25%) | **Yes** | **A** |
| 팀에 API·JSON을 무서워하지 않는 사람이 1명 이상 있다 | 개발자 1명. 단 B·C 응답에 *"워크플로우 툴 사용 경험 없음"* | **△** | B 또는 C (조건부) |
| "돌아가는 시스템"을 포트폴리오에 넣고 싶다 | 관련 응답 없음 | – | – |
| 마케팅·콘텐츠 직무에 관심이 있다 | "기획", "트렌드 분석" 응답 존재 | **△** | C |
| 기획·서비스 디자인 직무에 관심이 있다 | "프로젝트 리딩", "UI/UX 제작" 응답 존재 | **Yes** | **A** |
| **팀원 대부분이 비개발자다** | **75% (3/4)** | **Yes** | **A > C > B** |
| 팀원 일정이 자주 어긋난다 | 공통 시간대 확보됨 (§6 참조) | No | – |
| 요구사항이 모호한 게 싫고 체크리스트대로 하고 싶다 | 관련 응답 없음 | – | – |
| 눈에 확 띄는 발표 자료를 원한다 | 관련 응답 없음 | – | – |

**집계 결과: A 3표 / C 2표(△ 포함) / B 1표(△)**

### 가이드 §5 시나리오 매칭

> **시나리오 ①: "우리는 비개발자 위주다" → Project A 권장**
> *"API 연동 없이 이미지 생성 툴과 Figma만으로 완결됩니다. 실패 리스크가 가장 낮습니다."*

우리 팀은 **비개발자 75%** 로 이 시나리오에 정확히 부합합니다.

한편 시나리오 ④("개발 1~2명 + 비개발 다수 → C 권장")도 형식상 해당되지만, **C를 지지하는 근거가 무너지는 지점**이 있습니다. 시나리오 ④의 전제는 *"개발 인원이 워크플로우 골격을 잡고, 나머지가 프롬프트·콘텐츠·문서를 맡는 구조"* 인데, 우리 팀은 **프롬프트 제작과 노션 사용이 모두 '자신 없는 부분'으로 응답**되어 후방 지원이 성립하지 않습니다.

|---|---|:---:|

## 5. 평가 ③ — 가이드 §6 리스크 점검 (배점 20점)

가이드 §6-②의 리스크 점검 4문항을 설문으로 검증했습니다.

| 리스크 점검 문항 (가이드 §6) | 해당 | 설문 결과 | 판정 |
|---|:---:|---|:---:|
| Figma 계정과 기본 사용법을 아는 사람이 있는가? | A | **네 1명** (나머지 3명은 "처음 들어봄") | ⚠️ 통과(1인) |
| 노션 워크스페이스를 팀으로 쓸 수 있는가? | B·C | 주 사용자 **1명뿐** | ⚠️ 취약 |
| OpenAI API 키를 발급·결제할 사람이 있는가? | B·C | 개발자 1명 존재하나 **명시적 확답 없음** | ❓ 미확인 |
| 이미지 생성 툴 유료 크레딧을 낼 사람이 있는가? | A·C | **설문 문항 없음** | ❓ 미확인 |

### 단일 실패점(SPOF) 분석 — 이 평가서의 핵심

세 프로젝트 모두 특정 1명에게 기술이 몰립니다. 차이는 **그 사람이 막혔을 때 팀이 멈추는가**입니다.

| | **Project A** | **Project B / C** |
|---|---|---|
| 작업 구조 | **병렬** | **직렬** |
| 흐름 | 이미지 생성 · 후가공 · 로그 작성이 **동시 진행 가능**. Figma 배치는 마지막 단계 | 워크플로우가 연결돼야 → 결과가 나오고 → 그래야 저장·문서화 가능 |
| 1명이 막히면 | 나머지 3명은 계속 작업 가능 | **팀 전체 대기** |
| 가이드 §3 "망할 리스크" | **2점 (최저)** | B 8점 / C 6점 |

> 📌 가이드 §3의 해석 문단이 이 상황을 정확히 예고하고 있습니다.
> *"B: 가장 어렵지만 가장 명확합니다. … 대신 **API 연동이 막히면 팀 전체가 멈춥니다.**"*
> 자동화 툴 경험자가 0명인 우리 팀에서 이 리스크는 이론이 아니라 **높은 확률의 실제 시나리오**입니다.

| 프로젝트 | 리스크 대응력 점수 |
|---|:---:|
| **A** | **15 / 20** |
| B | 7 / 20 |
| C | 8 / 20 |
