# [Project A] AI 기반 UI/UX 디자인 시안 제작 — 결과물 해설서

> **한 줄 요약**
> "머릿속 기획을 → AI로 화면 이미지로 뽑고 → 다듬어서 → 클릭되는 프로토타입까지 만든다."

---

## 1. 이 미션의 본질부터 짚고 갑니다

미션 소개(§1)에 이런 문장이 있습니다.
> "단순히 예쁜 이미지를 만드는 법을 배우는 과정이 아닙니다."

즉, **평가 대상은 이미지의 예쁨이 아니라 '워크플로우'** 입니다.
`기획 → 프롬프트 → 이미지 생성 → 후가공 → 프로토타입`이라는 과정을 **증거로 남겼는가**가 핵심입니다.

이 관점을 잡고 나면 산출물이 왜 3개인지 자연스럽게 이해됩니다.

| 산출물 | 무엇을 증명하나 |
|---|---|
| UI 이미지 | 결과(Output)를 만들 수 있다 |
| 작업 로그 | 과정(Process)을 통제할 수 있다 |
| Figma 프로젝트 | 결과를 실무 산출물로 연결할 수 있다 |

---

## 2. 제출해야 하는 결과물 3종 (근거: §2 최종 결과물)

### 산출물 ①: UI 디자인 이미지 — **3장 이상**

**요구 조건 (§2, §4 기능 요구 사항)**

- 화면 **역할이 서로 구분**되어야 합니다. (메인 / 목록 / 상세 등)
- 이미지 내 **텍스트 깨짐이 수정된 상태**여야 합니다.
- 파일 형식은 **PNG 또는 JPG**.
- 비율은 **모바일 9:16** 또는 **데스크탑 16:9** 준수.

**예시 — '동네 러닝 크루 매칭 앱'을 만든다면**

| 파일명 | 화면 역할 | 담긴 요소 |
|---|---|---|
| `01_main_9x16.png` | 메인 | 오늘의 추천 크루 카드, 검색바, 하단 탭바 |
| `02_list_9x16.png` | 목록 | 지역별 크루 리스트, 필터 칩(거리/페이스/요일) |
| `03_detail_9x16.png` | 상세 | 크루 소개, 정기 러닝 일정, [참여 신청] 버튼 |

> 💡 **주의할 점**
> 같은 앱인데 1번은 파란 미니멀, 2번은 보라색 3D 스타일이면 감점 요인입니다.
> 과제 목표(§3)에 *"일관성을 유지하기 위한 방법(시드 고정, 이미지 레퍼런스 등)을 설명할 수 있다"* 가 명시되어 있기 때문입니다.
> → **시드(seed) 고정**, **스타일 레퍼런스 이미지 재사용**, **프롬프트 앞부분 고정** 같은 기법을 실제로 쓰고 그 사실을 로그에 적어두세요.

---

### 산출물 ②: 작업 로그 문서 — **1개**

**요구 조건 (§2, §4 프롬프트 최적화)**

- 사용한 프롬프트 전문 기록
- **초안 → 수정 → 최종**의 3단계 이상 변화 과정
- 각 단계에서 **"왜 바꿨는지"와 "결과가 어떻게 달라졌는지"**

미션에서 결과 예시(§7)로 제시한 표 형태를 그대로 따라가면 안전합니다.

**예시**

| 단계 | 프롬프트 | 변경 이유 | 결과 및 개선점 |
|---|---|---|---|
| 초안 | `running crew app UI, simple` | — | 웹인지 앱인지 불명확, 비율도 정사각형으로 나옴 |
| 수정 1 | `Mobile app UI design for running crew matching, clean minimal style, mint and white theme, bottom navigation bar, 9:16` | 플랫폼·비율·컬러를 명시해야 통제 가능하다고 판단 | 레이아웃 정상화. 다만 버튼 안 글자가 외계어로 깨짐 |
| 수정 2 | 위 프롬프트 + `--seed 12345` 고정 | 3개 화면의 스타일 통일이 필요 | 카드 모서리·그림자·폰트 톤이 3장 모두 일치 |
| 최종 | 위 프롬프트 + `high fidelity, Figma style, UI mockup` | 실사 사진풍이 아닌 '시안'처럼 보이게 하려고 | 시안 품질 확보. 깨진 텍스트는 Figma에서 텍스트 레이어로 덮어 최종 수정 |

> 💡 이 문서가 사실상 **채점 근거 문서**입니다.
> "예뻐졌다"가 아니라 **"어떤 단어를 넣었더니 무엇이 달라졌다"**로 써야 §3 과제 목표
> (*"프롬프트의 구성 요소가 결과물에 미치는 영향을 설명할 수 있다"*)를 충족합니다.

### 산출물 ③: Figma 프로젝트 — **1개 (링크)**

**요구 조건 (§2, §4 프로토타입 제작)**

- 생성 이미지를 배치해 **화면 흐름**을 구성
- **클릭 가능한 영역(Hotspot)** 또는 화면 전환 연결 표시
- **공유 링크** 형태로 제출 (권한을 "링크가 있는 모든 사용자 – 보기 가능"으로 열어두세요)

**예시 — 연결 구조**

```
[메인] --(크루 카드 탭)--> [목록] --(항목 탭)--> [상세]
   ^                                              |
   +---------------(뒤로가기 버튼)-----------------+
```

> 💡 Figma의 **Prototype 탭**에서 이미지 위에 투명 사각형을 올리고,
> 그 사각형에서 다음 프레임으로 화살표를 끌어 연결하면 Hotspot이 됩니다.

## 3. 후가공은 '선택'이 아니라 '필수'입니다 (근거: §4, §6 품질 기준)

제약 사항(§6)에 이렇게 못 박혀 있습니다.
> "AI가 생성한 이미지 속 뭉개진 글자나 부자연스러운 요소를 수정 없이 그대로 제출해서는 안 된다."

**현실적인 수정 방법 3가지**

1. **덮어쓰기 (가장 쉬움)** — Figma에서 깨진 글자 위에 사각형 + 실제 텍스트 레이어를 올립니다.
2. **인페인팅** — 해당 영역만 선택해 다시 생성 (DALL·E 편집, Photoshop 생성형 채우기 등).
3. **업스케일링** — 해상도가 낮아 뭉개진 경우 해상도를 키운 뒤 재작업.

## 4. 반드시 피해야 할 감점 요소 (근거: §6 제약 사항)

| 금지·필수 사항 | 구체적 행동 |
|---|---|
| 사용 툴 **이름 명시 필수** | "Midjourney v6 사용", "Figma 사용" 처럼 문서 첫 페이지에 기재 |
| Figma 외 도구 사용 시 | 도구명 + **사용 이유**를 함께 기재 |
| 레퍼런스 **출처 필수 기록** | "참고: Behance – OOO 프로젝트, URL" |
| Pinterest·Dribbble 캡처 제출 **엄격 금지** | 타인 작업물 캡처는 즉시 탈락 사유로 읽힙니다 |
| 저작권 해결된 툴 사용 | 무료/유료 무관하나 상업적 사용 조건을 확인 |

## 5. 보너스 과제 (근거: §5)

- **보너스 1 – 코드 변환 체험**: 완성된 시안 이미지를 v0, ChatGPT 등에 넣어 HTML/CSS로 변환해봅니다.
  → 변환된 화면 스크린샷 + 원본 시안 비교 이미지를 로그 문서 마지막에 붙이면 좋습니다.

---

## 6. 팀 작업 분배 & 40시간 운용 제안

| 역할 | 담당 업무 | 예상 시간 |
|---|---|---|
| 기획 리드 | 서비스 컨셉 정의, 화면 3개 역할 확정 | 6h |
| 프롬프트 엔지니어 | 프롬프트 실험 및 로그 문서 작성 | 14h |
| 이미지/후가공 | 생성·인페인팅·업스케일링 | 10h |
| 프로토타이퍼 | Figma 배치 및 Hotspot 연결 | 10h |

> 💡 화면 3개를 **서로 다른 사람이 각자 생성하면 스타일이 깨집니다.**
> 생성 담당은 1~2명으로 좁히고, 나머지는 후가공·문서·프로토타입에 배치하는 편이 안전합니다.

## 7. 최종 제출 전 체크리스트

- [ ] UI 이미지가 **3장 이상**이고 PNG/JPG인가?
- [ ] 세 화면의 **역할이 명확히 다른가**? (메인/목록/상세)
- [ ] 비율이 **9:16 또는 16:9**인가?
- [ ] 이미지 안에 **깨진 글자가 하나도 없는가**?
- [ ] 3장의 **스타일 톤이 일관**되는가?
- [ ] 작업 로그에 **초안 → 수정 → 최종**이 모두 있는가?
- [ ] 각 수정 단계에 **이유와 결과 차이**가 적혀 있는가?
- [ ] Figma 링크가 **외부에서 열리는가**?
- [ ] Figma에 **Hotspot / 화면 전환**이 표시되어 있는가?
- [ ] **사용 툴 이름**과 **레퍼런스 출처**가 문서에 명시되어 있는가?
