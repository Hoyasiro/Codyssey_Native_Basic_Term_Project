# [Project B] 뉴스 요약 자동화 워크플로우 만들기 — 결과물 해설서

> **한 줄 요약**
> "매일 정해진 시간에, 사람 손 없이 → RSS 뉴스를 긁어 → AI가 3줄로 요약해 → 노션에 차곡차곡 쌓는 시스템"

---

## 1. 이 미션의 본질부터 짚고 갑니다

미션 소개(§1)의 핵심 문장입니다.
> "단순히 하나의 기능을 구현하는 것이 아니라 스케줄 트리거와 주제 필터링 그리고 AI 요약과 에러 처리까지 자동화 파이프라인의 **전체 흐름을 설계하고 연결**하는 것이 핵심입니다."

즉 이 과제는 **"돌아가는 파이프라인"** 을 만드는 것입니다.
한 번 수동으로 눌러서 결과가 나오는 건 미달입니다. **사람이 자는 새벽에도 알아서 돌아야** 합니다.

세 프로젝트 중 **가장 '엔지니어링'에 가까운 과제**이며, 대신 요구 조건이 가장 구체적이라 **정답 경로가 명확**하다는 장점이 있습니다.

---

## 2. 제출해야 하는 결과물 3종 (근거: §2 최종 결과물)

### 산출물 ①: 자동화 워크플로우

**요구 조건**

- 매일 **설정된 시간에 자동 실행**되는 워크플로우
- 워크플로우 **구조를 보여주는 스크린샷**
- **각 단계별 역할과 연결 구조**를 설명하는 문서

**예시 — 미션 §7이 제시한 6단계 구조**

```
[1] 스케줄 트리거 (매일 09:00 / Asia/Seoul)
      ↓
[2] RSS 수집  ── 지정 RSS에서 최신 항목 가져오기
      ↓
[3] 주제 필터링 ── 키워드 'AI' 포함 항목 중 1건 선택
      ↓
[4] AI 요약   ── 프롬프트 적용, 3줄 요약 생성
      ↓
[5] 노션 DB 저장 ── Title / Text / URL / Date 매핑
      ↓
[6] 예외 처리 ── 실패 시 재시도(최대 2회) → 로그·스킵 → 알림
```

**단계별 설명 문서 작성 예시**

| 노드 | 도구 | 역할 | 실패 시 |
|---|---|---|---|
| Schedule | n8n Cron | 매일 09:00 KST 실행 | — |
| RSS Read | n8n RSS Feed Read | TechCrunch RSS 수집 | 빈 배열 → 워크플로우 종료 + 로그 |
| Filter | n8n IF | 제목에 `AI`/`인공지능` 포함 여부 | 미충족 시 스킵 |
| Summarize | OpenAI API | 3줄 요약 생성 | 최대 2회 재시도 |
| Notion | Notion API | DB에 페이지 생성 | 중복 키 존재 시 스킵 |

---

### 산출물 ②: 노션 데이터베이스 페이지

**요구 조건 (§2, §4 자동 저장)**

- AI가 요약한 텍스트가 포함된 뉴스 콘텐츠
- **제목, 요약문, 원문 링크, 발행일시가 각각 "올바른 속성"에 저장**된 상태

> ⚠️ 여기서 자주 실수합니다.
> 네 항목을 **한 덩어리 텍스트로 몰아넣으면 요구사항 미충족**입니다.
> 노션 DB의 **속성(Property) 타입**을 정확히 맞춰야 합니다.

**속성 설계 예시**

| 속성명 | 타입 | 들어가는 값 |
|---|---|---|
| 제목 | Title | AI 에이전트의 미래와 전망 |
| 요약 | Text | (3줄 요약문) |
| 원문 링크 | **URL** | https://example.com/news/123 |
| 발행일 | **Date** | 2026-01-20 |
| 중복방지키 | Text | 링크 해시 또는 GUID |

**저장 결과 예시 (§7)**

```
제목: AI 에이전트의 미래와 전망
요약:
  - 최신 AI 에이전트는 자율적인 의사결정 능력이 강화되고 있음
  - 기업 생산성 도구와의 결합 사례가 급증하는 추세임
  - 보안 및 윤리적 가이드라인의 필요성이 대두됨
원문 링크: https://example.com/news/123
발행일: 2026년 1월 20일
```

### 산출물 ③: README (또는 기획서)

**요구 조건 (§2)** — 이 세 가지가 **반드시** 들어가야 합니다.

1. **팀 역할 / 개인별 작업 요약**
2. **주제 필터링 기준(키워드·태그 목록)과 그것을 고른 이유**
3. **에러 처리 정책과 그것을 고른 이유**

> 💡 "이유"를 요구하는 이유가 있습니다.
> 과제 목표(§3)에 *"스케줄 트리거와 에러 처리를 포함한 **안정적인** 자동화 시스템의 **설계 원리**를 설명할 수 있다"* 가 있기 때문입니다.
> 결과만 있고 **판단 근거가 없으면 목표 미달**로 읽힙니다.

**README 작성 예시**

```markdown
## 주제 필터링 기준
- 키워드: AI, 인공지능, LLM, 생성형, 에이전트
- 선택 이유: 팀 관심 도메인이 생성형 AI이며, 일 평균 3~5건이 걸려
  "0건 수집"과 "과다 수집"을 모두 피할 수 있는 폭이라고 판단함.

## 에러 처리 정책
- RSS 0건 → 워크플로우 정상 종료 + Slack 알림 (실패로 간주하지 않음)
- OpenAI 타임아웃 → 최대 2회 재시도, 이후 스킵 + 로그
- 선택 이유: 무한 재시도는 API 비용이 누적되므로 §6 제약(비용 주의)에 따라 상한을 둠.
```

## 3. 놓치기 쉬운 세부 요구사항 (근거: §4)

| 항목 | 정확한 요구 |
|---|---|
| 수집 건수 | 필터 통과 항목 중 **1건 선택**해 제목·링크·본문 추출 |
| 요약 길이 | **3줄 이내** |
| 무인 실행 | 전 과정이 **사람 개입 없이** 완료 |
| 중복 방지 | 중복 저장 방지 키 **1개 이상 권장** (원문 링크 해시, GUID) |
| 호출 횟수 | 요약 호출은 **기사 1건당 1회 원칙** (재시도 제외) |
| 재시도 상한 | **최대 2회** |

> 💡 "3줄 이내"는 프롬프트에 강제해야 합니다.
> 예: `다음 기사를 한국어 3줄 이내로 요약하라. 각 줄은 40자 이내. 불릿 3개만 출력하고 다른 말은 하지 마라.`

## 4. 반드시 피해야 할 감점 요소 (근거: §6 제약 사항)

| 구분 | 내용 |
|---|---|
| 도구 | Make / Zapier / n8n 자유 선택, 단 **사용 툴 이름 명시** |
| AI 연동 | **OpenAI API 또는 동급 모델** 사용 |
| 저장소 | **노션 권장**, 타 도구 사용 시 명시 |
| 🚫 금지 | RSS 없는 사이트를 **강제 크롤링**하는 저작권 침해 방식 |
| 🚫 금지 | **무한 루프 트리거** (API 비용 폭탄 주의) |
| 🔐 보안 | API 키·토큰은 **환경 변수 또는 비공개 저장소**에 보관 |
| 🔐 보안 | 문서·스크린샷에 키가 노출되지 않도록 **마스킹 처리** |

> ⚠️ **스크린샷 제출 직전에 반드시 키 노출을 확인하세요.**
> n8n·Make의 노드 설정 화면을 그대로 캡처하면 토큰 일부가 찍히는 경우가 많습니다.
> 검은 박스로 덮은 뒤 제출하는 것이 안전합니다.

## 5. 보너스 과제 (근거: §5)

- **보너스 1 – 썸네일 자동 생성**: 이미지 생성 AI로 뉴스 주제에 맞는 썸네일을 만들어 노션에 함께 저장.
- **보너스 2 – 감성 분석 태그**: 기사의 긍정/부정을 분석해 노션 Select 속성으로 저장.
  → 요약 프롬프트에 `sentiment: positive|neutral|negative` 를 JSON으로 함께 뱉게 하면 **호출 1회 원칙**도 지킬 수 있습니다. (일석이조)

## 6. 팀 작업 분배 & 40시간 운용 제안

| 역할 | 담당 업무 | 예상 시간 |
|---|---|---|
| 워크플로우 빌더 | n8n/Make 노드 구성 및 연결 | 14h |
| 프롬프트 담당 | 3줄 요약 프롬프트 튜닝, 출력 형식 고정 | 6h |
| 노션 설계 | DB 속성 설계, 매핑 검증, 중복 방지 키 | 8h |
| 문서/QA | README 작성, 에러 시나리오 테스트, 마스킹 검수 | 12h |

> 💡 **최소 3일 이상 실제로 돌려보세요.**
> "매일 자동 실행"이 요구사항이므로, 노션에 **서로 다른 날짜의 레코드가 쌓인 스크린샷**이 가장 강력한 증거가 됩니다.
